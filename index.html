Continuous Integration в 2GIS Web API

Интеграционная сборка

Цель: проверить новую фичу и слить в мастер. Время выполнения должно быть 5–10 минут.

Что у нас есть

  • Ветка с новой фичей
  • Тесты
  • Система деплоя

Этапы интеграционной сборки

  1. Развернуть новую ветку
  2. Прогнать все тесты
    • Все хорошо — смержить ветку в мастер
    • Что-то упало — сообщить о проблеме

Сборка выполняется за 6 часов

  • Деплой за 5 минут
  • Выполнение тестов за 5 часов 55 минут

12 000 тестов

Анализ выполнения тестов

  • Максимальное время выполнения одного теста 2 минуты
  • Среднее время выполнения одного теста 1,5 секунды
  • Используется одно ядро (из 16) процессора при запуске PHPUnit

ParaTest

github.com/brianium/paratest
Поддержка параллельного выполнения тестов на PHPUnit.

Пример

			$ paratest -p 16 --phpunit ./phpunit \
  -c ./phpunit.xml ./tests

50 минут

Ищем компромисс

  • Долгая интеграция, но стабильный мастер
  • Быстрая интеграция, но возможные баги в мастере

Потери на этапе интеграции

  • Оставляем только приоритетные тесты (1500 тестов)
  • Оставляем только формат JSON (убрали JSONP и XML)

5 минут

Еще быстрее?

Переразворачивать полностью окружение избыточно когда правки только в приложении.

Деплой окружения и приложений

  • Мелкие правки с помощью Phing (только приложения)
  • Большие правки с помощью Chef (приложения + окружение)

Пример (Python)

			def update(issue, ...):
  if full_mode == 'yes':
    updater.update() // Chef
  else:
    updater.fast_update() // Phing

3 минуты

Боль

У нас автоматизированы тестирование и мерж фичи, а в JIRA по-прежнему пишем сами о результатах сборки.

Интеграция с JIRA

  • Комментарий в тикете об автоматическом мерже
  • Метка у тикета с веткой master

JIRA REST API Client

github.com/chobie/jira-api-restclient
API-клиент для JIRA на PHP.

Пример (PHP)

			$client->addComment($jiraIssue, $jiraComment);

$client->editIssue($jiraIssue, [
  'fields' => ['labels' => $labels]
]);

Интеграционная сборка в работе

			Feature Branch -> Deploy -> Priority Tests
  -> Merge to Master -> Mark Issue in JIRA

Профит

Быстрое решение об интеграции новой фичи и выявление критичных багов.

Но...

Мы все также ищем баги при подготовке релиза.

Регрессионная сборка

Регрессионная сборка

			Master Branch -> Deploy -> Full Tests

Регрессия перед релизом

			Master Branch -> RC -> Deploy -> Full Tests
  -> Merge to Stable -> Mark Issues in JIRA

Профит

Мастер без багов и готовность релизиться.

Как так?!

Упала миграция на бою, хотя регрессия прошла успешно.

Задача
